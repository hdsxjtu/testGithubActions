## 1. 实时操作系统 (RTOS) 与并发基础 (深度扩展)

**面试高频考点**：强调“实时性”（确定性），而非“速度快”。系统必须在规定的死区时间（Deadline）内做出确定性的响应。

*   **任务间通信方式大全 (全集)**：
    1.  **消息队列 (Message Queue)**：传递不定长数据，支持FIFO，可传值或传指针。
    2.  **信号量 (Semaphore)**：用于任务同步（二值信号量）或资源计数（计数信号量）。
    3.  **互斥量 (Mutex)**：专用于保护共享资源，自带优先级继承机制。
    4.  **事件标志组 (Event Group)**：用于多个事件的同步（比如任务C需要等待任务A和任务B都完成后才执行）。
    5.  **任务通知 (Task Notification)**：FreeRTOS特有，比信号量更快、占用内存更小，直接向指定任务发送信号。
    6.  **邮箱 (Mailbox)**：通常指长度为1的消息队列，适合传递单个状态。

*   **二值信号量 vs 互斥锁 (Mutex) 的核心区别**：
    *   **用途不同**：信号量用于**同步**（A触发中断，B处理），互斥锁用于**互斥**（A和B都要用串口，谁先拿到谁用，用完归还）。
    *   **归还要求不同**：信号量可以由中断给出，由任务获取；互斥锁**必须由获取它的任务来释放**（谁污染谁治理），不能在中断中使用互斥锁。
    *   **优先级继承**：互斥锁**有**优先级继承机制，二值信号量**没有**。

*   **🔥 深入解析：优先级翻转 (Priority Inversion) 的 3 个角色流程**：
    *   **前置条件**：存在三个任务：高优先级(H)、中优先级(M)、低优先级(L)。H 和 L 共享一个资源，且由信号量保护。
    *   **流程**：
        1. L 先运行，获取了信号量，开始使用共享资源。
        2. H 被唤醒，抢占 L，尝试获取信号量。由于信号量在 L 手里，H 被阻塞挂起。
        3. 此时，M 任务（不需要该资源）就绪，由于 M 优先级高于 L，M 抢占了 L，开始一直运行。
    *   **结果**：H 必须等待 L 释放资源，但 L 被 M 剥夺了 CPU 无法运行。导致最高优先级的 H 被中优先级的 M 变相阻塞了！这就是优先级翻转。
    *   **解决机制（互斥锁的优先级继承）**：当 H 尝试获取 L 持有的互斥锁被阻塞时，RTOS 临时将 L 的优先级提升到与 H 相同级别。这样 M 就无法抢占 L，L 能快速执行完并释放锁，随后 H 获取锁继续运行，L 恢复原有低优先级。

*   **多任务数据竞争 (Data Race) 原理与举例**：
    *   **原理**：两个或多个任务（或中断）在未加同步保护的情况下，同时对同一个全局变量或寄存器进行读写-修改-写回（RMW）操作，导致数据错乱。
    *   **举例**：假设全局变量 `count = 0`。任务A 执行 `count++`，任务B也执行 `count++`。
        1. 任务A读取 `count` (值为0) 到 CPU 寄存器 R1。
        2. 此时发生任务切换，任务B打断A。任务B读取 `count` (值为0) 到 R2，执行加1变为1，写回 `count`。此时内存中 `count=1`。
        3. 切回任务A，任务A继续执行 R1 加1变为1，写回 `count`。
        4. 最终结果：执行了两次 `count++`，但 `count` 的值变成了 1 而不是 2。
    *   **解决**：临界区保护（关中断）、互斥锁、或者使用原子操作。

*   **Bootloader 跳转写 Flash 为什么要关闭中断？**
    *   **原因 1（向量表未切换）**：在擦写 Flash 期间，如果发生中断，PC 指针会去中断向量表找中断服务函数(ISR)。如果此时新的 APP 还没搬运完或向量表没重定向，PC 就会跳到一个非法的或正在被擦除的 Flash 地址执行，导致**系统跑飞 (HardFault)**。
    *   **原因 2（Flash 忙碌）**：MCU 在擦写内部 Flash 时，Flash 总线通常是阻塞的或处于特殊状态。如果 ISR 也在 Flash 中，此时去读取 ISR 指令会导致总线错误。因此必须关中断（`__disable_irq()`），直到跳转到新 APP 并且重设了向量表（VTOR寄存器）后再打开。

## 2. 内存管理与疑难杂症排查 (Memory & Debugging)

*   **RAM 中的段划分 (BSS, DATA, 栈, 堆)**：
    *   **`.bss` 段**：存放**未初始化**的全局变量和静态变量。启动代码会将其清零。
    *   **`.data` 段**：存放**已初始化**的全局变量和静态变量。启动时从 Flash 拷贝初值到 RAM。
    *   **Heap (堆)**：用于 `malloc()` / `free()` 动态分配。生长方向一般向上（低地址向高地址）。
    *   **Stack (栈)**：用于局部变量、函数参数、返回地址、中断现场保存。生长方向一般向下（高地址向低地址）。
    *   **程序段 (`.text` / `.rodata`) 在哪？**：通常在 **Flash (ROM)** 里直接执行（XIP: Execute In Place），不占用 RAM。但在某些追求极致性能的芯片中，会把部分关键代码（如中断函数）从 Flash 搬到 RAM（ITCM/SRAM）中执行。

*   **任务的堆栈大小 (Stack Size) 如何确认？多大合适？**
    *   **方法 1 (经验预估)**：看任务里有没有定义巨大的局部数组、有没有极深的函数嵌套调用、有没有使用 `printf` (极耗栈)。
    *   **方法 2 (RTOS 监控工具 - 面试官最想听的“还有吗”)**：
        1.  开启 FreeRTOS 的 `uxTaskGetStackHighWaterMark()` 函数，它可以返回任务运行历史中，剩余堆栈的最小历史值（高水位线）。如果这个值接近 0，说明危险；如果很大，说明浪费，可以缩小。
        2.  开启堆栈溢出钩子函数 `vApplicationStackOverflowHook()`，一旦溢出系统会自动跳入这个死循环，方便打断点定位。
        3.  使用 IDE 插件（如 IAR/Keil 的 RTOS 插件，或 SEGGER SystemView）实时可视化查看每个任务的栈使用率。
        4.  在栈的末尾填充魔术字（Magic Pattern 比如 `0xDEADBEEF`），定期检查这块内存有没有被改写。

*   **系统跑飞 (HardFault) 如何检查？**
    *   **易复现的跑飞**：
        *   **挂载调试器**：在线仿真，等系统跑飞进入 `HardFault_Handler` 时暂停。
        *   **查看现场**：通过看 LR 寄存器找到导致异常的那条汇编指令的地址，或者查看调用栈 (Call Stack) 回溯上层函数。
        *   **查看内核寄存器**：通过 Cortex-M 的 SCB->CFSR (可配置错误状态寄存器) 查看具体是内存越界 (MemManage)、总线错误 (BusFault) 还是用法错误 (UsageFault，如除以0)。
    *   **不易复现的跑飞 (偶发性)**：
        *   **记录 Log (日志)**：把 `HardFault_Handler` 改造一下，在跑飞瞬间，将当前的 PC 指针、LR、R0-R3 等核心寄存器信息保存到后备寄存器 (RTC Backup RAM) 或直接写入 EEPROM/Flash 尾部。等下次板子重启重启时打印出来。
        *   **软件看门狗 (Watchdog)**：确保跑飞后能自我恢复，不影响现场业务。

*   **为什么 Linux 排查跑飞比 MCU 方便？**
    *   **内存管理单元 (MMU)**：Linux 有强大的 MMU 支持，每个进程有独立的虚拟地址空间。一个进程如果访问了野指针或发生段错误 (Segfault)，MMU 会直接拦截并杀死该进程，生成 Core Dump (核心转储) 文件，**绝对不会影响整个系统崩溃**。
    *   **丰富的排查工具**：拿到 Core Dump 文件后，直接用 `gdb` 加载就能完美还原案发现场的所有变量和调用栈。还可以用 `Valgrind` / `ASAN` 轻松查出内存泄漏。而 MCU 通常是扁平化内存，一个任务踩了野指针，可能把其他任务或操作系统的核心数据破坏了，导致最终跑飞的地方根本不是案发第一现场，排查极度困难。

## 3. 采集与常用滤波算法

在计量 (Metering) 场景中，ADC 采集电压电流常伴随噪声，必须进行软件滤波。

* **滑动窗口滤波 (Moving Average Filter)**：

  * **原理**：开辟一个长度为 N 的数组（FIFO 环形缓冲区），每次采入新值剔除最老的值，求平均。

  * **应用**：对周期性干扰有良好的抑制作用，平滑度高。缺点是会带来相位延迟。

* **一阶低通滤波 (IIR 1st-order Low-pass Filter)**：

  * **公式**：$Y\_n = \alpha \cdot X\_n + (1 - \alpha) \cdot Y\_{n-1}$ （其中 $\alpha$ 是滤波系数，0\~1之间）。

  * **应用**：滤除高频毛刺。系数 $\alpha$ 越小，滤波效果越平滑，但响应越慢。占用内存极小（只需保存上一次的结果）。

* **中值滤波 (Median Filter)**：

  * **原理**：连续采样 N 次，排序后取中间值。

  * **应用**：极度克制**脉冲干扰**（例如突然一个极大的尖峰毛刺，会被直接剔除）。

* **卡尔曼滤波 (Kalman Filter)**：

  * **原理**：基于预测与更新机制的最优化自回归数据处理算法。它结合了系统的预测模型和传感器的实际测量值，通过动态计算“卡尔曼增益”来决定更信任哪一个，从而得出最优估计。
  * **应用**：在嵌入式中表现极佳。不仅能用于多传感器数据融合（如陀螺仪+加速度计解算姿态），即使降维用于一维的电压/电流/ADC采集，也能在保留信号快速跳变能力的同时，完美滤除高斯白噪声。

* **一阶互补滤波 (Complementary Filter)**：

  * **原理**：巧妙地结合高通与低通滤波器。公式：$Y_n = \alpha \cdot (Y_{n-1} + \Delta X_{high}) + (1 - \alpha) \cdot X_{low}$。
  * **应用**：极其经典的姿态解算算法。由于加速度计求角度容易受低频震动干扰（适合低通），而陀螺仪求角度会有随时间累积的高频漂移（适合高通）。互补滤波将两者结合，以极低的计算量（远低于卡尔曼）得到非常平滑且不滞后的结果，非常适合算力有限的单片机。

## 4 & 5. 软件模块化设计与代码质量

* **模块化设计核心**：**高内聚，低耦合 (High Cohesion, Low Coupling)**。

* **分层架构**：

  * HAL 层（硬件抽象）：操作底层寄存器。

  * Driver 层：具体外设驱动（如 EEPROM, LCD）。

  * App 层：业务逻辑。App 层绝不能直接包含底层寄存器头文件，必须通过接口函数调用。

* **解耦技巧**：多使用**回调函数 (Callback / 函数指针)**。底层发生中断后，通过回调通知高层，而不直接调用高层函数。

* **代码规范**：

  * 遵循 **MISRA C** 标准（汽车及工业级 C 语言安全子集，例如：禁用 `goto`，禁用动态内存分配 `malloc` 等）。

  * 严格的命名规范，结构体使用 `typedef`。

* **代码质量保证**：Code Review（代码审查），编写单元测试（Unity, CppuTest），使用静态分析工具。

## 6. CI/CD 与 DevOps (参考我们之前的实战)

* **CI (持续集成)**：通过 GitHub Actions，每次 `push` 代码都会自动触发一台干净的服务器执行 `make` 编译，确保代码能编译通过，不破坏主分支。

* **🔥 核心难点：如何在 CI 云端服务器（Linux）中运行单片机单元测试？**
  * **难点**：云端没有 STM32 硬件，无法执行读写外设寄存器的代码。
  * **解法 1：基于主机的测试 (Host-based Testing) + Mock (打桩)**
    * 使用测试框架（如 Unity, CppuTest）和 Mock 工具（如 CMock, fff）。
    * 将业务逻辑和底层 HAL 库解耦。在测试时，利用 Mock 工具生成一个“假”的硬件操作函数（例如遇到 `HAL_GPIO_WritePin` 直接返回成功，而不是去操作真实物理寄存器）。
    * 在 GitHub Actions 中，**放弃使用交叉编译器** (`arm-none-eabi-gcc`)，而是直接使用 Linux 平台的标准 `gcc`，将业务逻辑与 Mock 桩函数一起编译成 Linux 上的可执行文件直接运行断言。
  * **解法 2：硬件在环测试 (HIL, Hardware-In-the-Loop)**
    * 若必须依赖真实硬件，可以使用 **Self-hosted Runner (自托管运行器)**。将实验室里一台连着真机开发板和 J-Link 的本地电脑挂载到 GitHub。流水线触发时，指令发给实验室电脑，自动烧录固件并通过串口提取测试结果。
  * **覆盖率与拦截机制**：流水线执行测试程序后，使用 `gcov` / `lcov` 自动生成代码覆盖率 (Code Coverage) 报告。如果有任何断言失败 (Exit code != 0)，流水线变红，自动拦截代码合并。

* **静态代码扫描**：例如我们使用的 **CodeQL** 或 **Cppcheck**，在不运行代码的情况下，通过数据流分析出内存越界、死循环、空指针等漏洞。

* **二进制扫描 (SCA/Binary Analysis)**：分析编译好的 `.bin` 或 `.hex` 文件，检查是否引用了有已知漏洞的第三方开源库，或者是否包含明文密码（Hardcoded Secrets）。

* **CD (持续交付)**：通过打 `Tag` (如 v1.0)，流水线自动打包生成固件，并发布到 Release 页面。

## 7. 嵌入式网络安全 (Cybersecurity) 🔥 核心加分项

* **威胁模型 (Threat Modeling)**：常用 **STRIDE** 模型（Spoofing 欺骗, Tampering 篡改, Repudiation 抵赖, Info Disclosure 信息泄露, DoS 拒绝服务, EoP 提权）。分析系统从哪个物理接口或协议可能被黑客入侵。

* **对称加密 vs 非对称加密**：

  * **对称加密 (AES)**：加密解密用同一把钥匙。速度快，适合大批量数据（如加密固件包）。

  * **非对称加密 (RSA/ECC)**：公钥+私钥。公钥加密私钥解，或私钥签名公钥验。速度慢，适合秘钥交换和身份认证。

* **哈希 (Hash, SHA-256)**：提取数据指纹，不可逆。用于校验固件**完整性**（防止被篡改）。

* **签名与证书 (Signature & Certificate)**：**安全启动 (Secure Boot)** 的核心！

  * **签名** = 固件 Hash 值 + 开发者私钥加密。

  * 设备端用开发者公钥解密签名，比对 Hash 值。确保固件既**没被篡改**，也**确实是官方发布的**。

* **单片机读写保护 (RDP/WRP)**：

  * STM32 的 RDP (Readout Protection) 分 Level 0, 1, 2。

  * Level 1 防止调试器 (J-Link) 读取 Flash 代码，但允许代码自己读自己；Level 2 彻底锁死芯片（无法降级或重刷）。防止黑客通过 SWD 接口扒取二进制固件反编译。

## 8 & 9. 敏捷开发 (Agile) 与 AI 工具应用

* **敏捷开发 (Scrum)**：

  * 不写上百页的死板需求文档，而是将任务拆分为 Sprint（迭代周期，通常 2-3 周）。

  * **Daily Standup（每日站会）**：昨天做了什么，今天做什么，遇到什么阻塞（Blocker）。

  * 使用 Jira 管理看板（To Do, In Progress, Review, Done）。

* **AI 工具 (GitHub Copilot)**：

  * **日常用法**：自动补全枯燥的 `switch-case` 和寄存器配置；根据函数名自动生成注释代码；编写冗长的单元测试用例。

  * **进阶用法**：遇到生僻的编译报错，直接高亮报错代码让 Copilot 解释；要求 AI 检查某段 C 代码是否存在缓冲区溢出风险。

## 10 & 11. Python 与 工业通信协议

* **Python 的作用**：在嵌入式开发中，Python 是神级辅助工具。

  * 编写自动化测试脚本（使用 `pyserial` 库通过串口发指令，自动验证单片机响应）。

  * 数据分析（使用 `matplotlib` 将 ADC 采集到的电流电压数据画成波形图，验证滤波算法效果）。

  * 编写打包脚本（将 `.bin` 文件增加 CRC 校验头）。

* **Modbus 协议**：

  * 工业界最广泛的协议。主从架构（Master/Slave），一条总线只能有一个主机发起请求。

  * 分类：Modbus RTU（基于串口 RS485，数据为二进制，靠时间间隔分帧）和 Modbus TCP（基于以太网）。

  * 四大寄存器：线圈（读写位）、离散输入（只读位）、输入寄存器（只读 16-bit）、保持寄存器（读写 16-bit）。

* **CAN 与 CANopen**：

  * **CAN**：差分信号，抗干扰强。多主机，基于 ID 的**非破坏性仲裁**（ID越小，优先级越高，同时发数据时 ID 小的抢占总线）。

  * **CANopen**：CAN 只是物理层和数据链路层，CANopen 是上面的应用层。它引入了 **对象字典 (Object Dictionary, OD)**。

  * **PDO (过程数据对象)**：用于实时传输周期性数据（如传感器数据），不需确认，速度快。

  * **SDO (服务数据对象)**：用于非实时的参数配置（如修改单片机配置参数），带应答机制。

## 12. 面试高光项目展示：本质安全感烟/感温探测器

*(建议采用 STAR 法则：情境 Situation - 任务 Task - 行动 Action - 结果 Result 进行阐述)*

### 🎙️ 1. 一分钟核心话术 (STAR)

*   **【情境与背景 (Situation)】**
    > “我主导开发了一款**本质安全的感烟和感温探测器**。这款产品主要用于高危工业场景，对可靠性要求极其苛刻。而且产品是面向大规模量产的，**年出货量在 5 万台以上**，一旦出现软件 Bug 导致返工，召回成本将是毁灭性的。”

*   **【任务与挑战 (Task)】**
    > “我的核心任务不仅是实现基础的传感采集，更重要的是解决三大挑战：一是保证总线通信的绝对稳定；二是降低硬件成本的同时保证配置数据的掉电安全；三是设计一套绝对可靠的现场升级方案。”

*   **【我的核心行动与技术深度 (Action)】**
    > “针对这三大挑战，我做了如下设计：
    > 1. **通信高可靠**：通过底层的硬件 DMA 结合 RTOS 队列，外加应用层的确认重传和 CRC 校验机制，做到了在恶劣电磁环境下，**百万次连续通信不丢一帧数据**。
    > 2. **数据存储优化**：为了省去外部 EEPROM 的成本，我在内部 Flash 实现了**模拟 EEPROM 算法**，加入了磨损均衡（Wear Leveling）和双页交换机制，保证了配置参数在频繁掉电下绝不丢失。
    > 3. **无感且安全的升级**：设计了一套带 **双 Slot（A/B 面）切换机制的 Bootloader**。新固件先在后台下载到 Slot B，校验完整性后，再由 Bootloader 进行无缝切换。即使在升级断电的极端情况下，也能安全回退，做到了真正的‘防变砖’。”

*   **【最终结果 (Result)】**
    > “最终，产品成功量产，经受住了大规模市场检验，通信稳定性成了我们产品最大的卖点之一，且从未发生过因为升级导致的返厂事故。”

### ⚠️ 2. 面试官“夺命连环追问”防守指南

*   **追问 1：“百万次通信不丢帧，你是怎么证明的？底层具体是怎么处理通信的？”**
    *   **防守策略**：绝不能说“在中断里直接处理数据”。
    *   **回答**：“我们搭建了自动化拷机平台，连续几天几夜高频发包并比对序号。底层使用了 **DMA 配合双缓冲区（Ping-Pong Buffer）** 接收数据。中断只负责抛出信号量，真正的数据解析交由 RTOS 的高优先级任务完成，应用层加入了**严格的超时机制与应答重传（ACK）**。”

*   **追问 2：“Flash 模拟 EEPROM，如果正在写入的那一瞬间突然断电了，数据会坏吗？”**
    *   **防守策略**：这是必考题！必须抛出 **Page Swapping (页交换)** 和 **魔术字 (Magic Word)** 概念。
    *   **回答**：“分配了 Page A 和 B。平时追加写入 A，满时将最新状态拷贝到 B，并在 B 尾部打上一个‘Valid’的魔术字，最后擦除 A。如果在拷贝途中断电，B 尾部没有魔术字，重启后系统会丢弃 B，继续使用 A 的旧数据。**这样最多丢失最后一次修改，绝不会导致整个参数表损坏。**”

*   **追问 3：“双 Slot 升级，如果 Bootloader 把 B 拷贝到 A 的中途断电了，不就变砖了吗？”**
    *   **防守策略**：考察 Bootloader 的极端异常处理逻辑。
    *   **回答**：“我们在拷贝前，会给 A 区的原始状态做一个‘正在覆盖’的备份标志。如果拷贝中途断电，Bootloader 重启后会发现 A 区 Checksum 失败且带有未完成标志，此时它会**触发恢复机制**，重新从 B 区完整拷贝一次，直到 A 区校验通过才去执行。所以永远不会变砖。”

*   **追问 4：“‘本质安全’对你的软件开发有什么特殊要求？”**
    *   **防守策略**：本安电路特点是严格限流防电火花。
    *   **回答**：“软件层面最重要的配合是**极度的低功耗控制**和**防电压跌落**。必须避免外设同时全开导致功耗峰值超标。此外，为了应对本安限流可能导致的电压突然波动，软件里做了严格的 PVD（可编程电压检测）中断抢占，一旦跌落立刻强制保存关键数据。”
